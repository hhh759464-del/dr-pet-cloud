### 2.2 路由设计

| 路径 | 页面 | 权限 | 说明 |
|------|------|------|------|
| `/auth` | AuthView | Public | 登录/注册，已登录自动跳转 |
| `/pets` | PetsView | Auth | 宠物列表首页 |
| `/standby/:petId` | StandbyView | Auth | 待机守护（开启前） |
| `/guarding/:petId` | GuardingView | Auth | 守护中（核心功能） |
| `/report/:sessionId` | ReportView | Auth | 守护报告（停止后） |
| `/insights/:petId` | InsightsView | Auth | 情绪深度分析 |
| `/voices/:petId` | VoicesView | Auth | 语音录制管理 |
| `/settings/:petId` | SettingsView | Auth | 守护参数设置 |
| `/` | - | - | 重定向到 `/pets` |

**路由守卫逻辑：**
```
router.beforeEach(to):
  session = supabase.auth.getSession()
  if to.requiresAuth && !session → redirect /auth
  if to.path == /auth && session → redirect /pets
```

### 2.4 状态管理

本项目 **不使用 Pinia/Vuex**，而是使用 **Composables（组合式 API）** 管理状态，原因：
- MVP 阶段数据流简单，不需要全局状态库
- `useAuth` 提供用户状态，`useAudio` 提供音频引擎状态
- 页面间通过路由参数（`:petId`、`:sessionId`）传递数据
- 持久化数据通过 Supabase 和 localStorage

```
useAuth.js:
  state: user (ref), loading (ref)
  methods: signUp(), signIn(), signOut()
  lifecycle: onMounted → getSession → onAuthStateChange

useAudio.js:
  state: isListening, currentDb, isTriggered, timeData, frequencyData
  methods: startListening(), stopListening(), setOnTrigger(), setOnDbUpdate()
  lifecycle: requestAnimationFrame loop → AnalyserNode → RMS → 触发逻辑
```

## 3. 音频处理引擎

### 3.1 数据流

```
麦克风硬件
  │
  ▼
getUserMedia({ audio: true })
  │
  ▼
MediaStreamSource ──→ AnalyserNode (FFT 256)
  │                        │
  │                        ├── getByteTimeDomainData → timeData[]
  │                        │       │
  │                        │       ▼
  │                        │   RMS 计算: √(Σ(normalized²) / N)
  │                        │       │
  │                        │       ▼
  │                        │   dB 近似: 20*log10(rms) + 94
  │                        │       │
  │                        │       ├── → Waveform Canvas (实时绘制)
  │                        │       └── → 触发逻辑判断
  │                        │
  │                        └── getByteFrequencyData → frequencyData[]
  │                                (预留给后续频率分析)
  │
  ▼
requestAnimationFrame(loop)
  └── 每帧更新 currentDb → 触发判断 → 安抚/冷却
```

### 3.2 触发状态机

```
         ┌──────────┐
         │  IDLE    │ ← 初始状态 / 冷却结束
         └────┬─────┘
              │ dB >= THRESHOLD
              ▼
         ┌──────────┐
         │  LISTEN  │ ← 持续监听（接近阈值但未达时长）
         └────┬─────┘
              │ 持续 >= TRIGGER_DURATION
              ▼
         ┌──────────┐
         │ TRIGGERED│ ← 触发焦虑事件
         └────┬─────┘
              │ 播放安抚语音
              ▼
         ┌──────────┐
         │ COOLDOWN │ ← 冷却中（COOLDOWN_MS 内不触发）
         └────┬─────┘
              │ 冷却结束
              ▼
         ┌──────────┐
         │  IDLE    │ ← 回到监听
         └──────────┘
```

### 3.3 关键参数

```javascript
// src/composables/useAudio.js
const THRESHOLD_DB = 75       // 默认 75dB，可配置 40-100
const TRIGGER_DURATION_MS = 2000  // 默认 2s，可配置 1-5s
const COOLDOWN_MS = 60000     // 默认 60s，可配置 30-300s
const FFT_SIZE = 256          // 128 个频率区间
```

### 4.2 数据库表关系

```
auth.users (Supabase 内置)
    │
    │ 1:N
    ▼
pets ─────────────┐
    │              │
    │ 1:N          │ 1:N
    ▼              ▼
soothing_voices   guard_sessions
    │              │
    │ 1:N          │ 1:N
    ▼              ▼
(Storage)         anxiety_events

pets ───────────── daily_reports (UNIQUE pet_id + date)
```

### 4.3 RLS（行级安全）策略

```sql
-- pets: 用户只能操作自己的宠物
CREATE POLICY "Users manage own pets" ON pets
  FOR ALL USING (auth.uid() = owner_id);

-- soothing_voices: 通过宠物关联
CREATE POLICY "Users manage own voices" ON soothing_voices
  FOR ALL USING (EXISTS (
    SELECT 1 FROM pets
    WHERE pets.id = soothing_voices.pet_id
    AND pets.owner_id = auth.uid()
  ));

-- guard_sessions, anxiety_events, daily_reports 同理，层层关联
```

### 4.4 Storage 方案

| 存储内容 | 方案 | 格式 |
|---------|------|------|
| 安抚语音文件 | Supabase Storage (`soothing-voices` 桶) | WebM |
| 情绪事件数据 | Supabase PostgreSQL | JSON |
| 用户设置 | localStorage | JSON |
| 运行时音频 | 内存（不持久化） | Uint8Array |

Body: {
  model: "deepseek-chat",

### 5.2 降级策略

```
DeepSeek API 可用？
  ├── 是 → 返回 AI 生成的个性化总结
  └── 否 → 调用 generateFallbackSummary(stats)
            → 基于规则模板拼接待办文字
            → 确保用户始终能看到报告
```

## 6. 安全设计

### 6.1 认证安全
- Supabase Auth JWT，自动 refresh token
- 密码最小长度 6 位
- 邮箱格式前端验证

### 6.2 数据安全
- RLS 策略确保用户数据隔离
- 前端 API Key（anon key）为公开密钥，不暴露 service_role key
- DeepSeek API Key 通过 Vite 环境变量注入，构建时替换

### 6.3 隐私安全
- 麦克风权限仅在"开启守护"时请求，停止守护时释放
- 录音文件仅存储在用户自己的 Supabase 项目中
- 无用户行为追踪/埋点（MVP 阶段）
