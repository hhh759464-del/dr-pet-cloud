## 1. 产品概述

### 1.1 产品愿景

**让每一个宠物主人在外出时，都能通过闲置设备给宠物一个"虚拟主人"的陪伴。**

### 1.2 产品定位

宠博士·云伴（Dr. Pet · Cloud Companion）是一款面向分离焦虑宠物的 Web 端智能安抚应用。它利用闲置手机/电脑的 Web Audio API 进行环境音实时监听，当检测到宠物吠叫/呜咽等焦虑行为时，自动播放主人预录的安抚语音，并基于 DeepSeek 大模型生成每日情绪报告。

### 1.3 一句话电梯演讲

> 对于外出工作时担心宠物分离焦虑的上班族，宠博士·云伴是一款基于 Web Audio API 的智能安抚 App，它能自动检测宠物叫声并播放主人语音进行安抚，而市面上宠物摄像头仅支持被动观看且价格昂贵（¥200-500），我们的产品只需闲置设备浏览器即可运行，零硬件成本，还能生成 AI 情绪日报。

### 1.4 核心价值主张

- **零门槛：** 浏览器打开即用，无需购买硬件、无需安装 App
- **真安抚：** 不是冰冷的监控画面，而是主人真实声音的情感介入
- **看得懂：** AI 情绪日报，把抽象的声音数据变成可读的情感总结
- **多人多宠：** 支持家庭多人使用，多只宠物独立档案

## 2. 市场背景与竞品分析

### 2.1 行业背景

- 2025 年中国城镇宠物（犬猫）消费市场规模达 3000+ 亿元
- 中国城镇养宠人数超过 7000 万，其中独居/合租上班族占比超 60%
- 分离焦虑是城市宠物最高发的行为问题之一，约 40% 的宠物犬存在不同程度分离焦虑
- 现有宠物智能设备渗透率不足 15%，门槛主要在于价格和安装复杂度

### 2.2 竞品分析

| 竞品 | 类型 | 优势 | 劣势 | 我们的应对 |
|------|------|------|------|-----------|
| **Furbo** | 宠物摄像头+零食投喂 | 实时视频、双向通话、零食投喂、硬件稳定 | ¥500-1000，手动喊话依赖主人在线 | 砍掉视频和硬件，只做"自动安抚"这一个点 |
| **小米宠物摄像头** | 监控摄像头 | 性价比高（¥199）、米家生态联动 | 被动观看 + 手动喊话，无检测触发逻辑 | 价格我们更低（免费），行为上我们更主动 |
| **PetCube** | 宠物摄像头+激光 | 激光互动、远程逗宠 | ¥300-600，售后不便，同样依赖手动 | 激光互动不能安抚分离焦虑，语音才能 |
| **DogTV** | 内容播放 | 科学配乐，持续播放 | 无检测触发，宠物对固定内容会脱敏 | 我们是"检测到才播放"，不是"一直放" |

**关键认知**：所有摄像头产品的安抚逻辑都是——

```
狗叫 → 推送通知给主人 → 主人打开App → 按住说话 → 安抚
```

这个链路的前提是**主人实时在线**。但主人的真实状态是：在开会、在开车、在睡觉、在地铁上没信号。我们的核心差异只有一句话：**把"手动安抚"变成了"自动安抚"。**

### 2.3 差异化定位（诚实版）

| 维度 | 传统宠物摄像头 | 宠博士·云伴 | 说实话 |
|------|--------------|-----------|--------|
| 实时视频 | 有 | 无 | **我们输了。** 摄像头在这一点无法被替代 |
| 安抚方式 | 手动喊话 | 自动检测→触发→播放 | **我们赢了。** 这是唯一的护城河 |
| 硬件成本 | ¥199-1000 | ¥0（复用闲置设备） | 赢在价格，但闲置设备也是一种隐性成本 |
| 硬件稳定性 | 插电专用设备，不掉线 | 旧手机/浏览器可能崩溃 | **我们输了。** 这是产品体验的最大短板 |
| 数据报告 | 时间线截图 | AI 情绪日报+趋势分析 | 赢了，但这是软件层面的优势，摄像头也能加 |
| 安装难度 | 需配网/打孔/插电 | 浏览器打开即用 | 赢在初始门槛 |

### 2.4 面试必答题：如果 Furbo 加一个"自动播放语音"功能呢？

**诚实回答：技术上就是一次固件更新。Furbo 有麦克风有喇叭，想加这个功能几天就能上线。**

所以我们不能赌"竞品不会做"，而是要问：**竞品做了之后，我们还有什么？**

到那时，我们的防御不在功能，在三件事：

| 防御层 | 说明 | 竞品复制的难度 |
|------|------|-------------|
| **价格** | 我们永远免费（或极低订阅价），摄像头永远需要硬件成本 | 高——硬件成本无法归零 |
| **行为数据** | 长期积累的宠物情绪数据，能训练出个性化安抚策略（如"这只柯基对女主人的声音反应更好"） | 中——需要时间积累，不是功能开发能追上的 |
| **多设备家庭** | 一个 App 管多个旧手机 + 多只宠物 + 多个房间，摄像头用户做不到（买 3 个 Furbo 要 ¥1500+） | 中——硬件成本壁垒 |

### 2.5 我们在哪些场景下会输

诚实列出竞品更优的场景，避免面试时被追问到死角：

| 场景 | 谁更好 | 原因 |
|------|--------|------|
| 主人想实时看狗狗在干嘛 | 摄像头胜 | 我们有实时音频波形，但没有视频 |
| 主人想和狗互动（喊话、投零食） | 摄像头胜 | 我们是单向的（检测→播放），做不到双向 |
| 主人想确认狗叫了再手动安抚 | 摄像头胜 | 有些用户就是想自己控制 |
| 主人开会/开车/睡觉时狗叫了 | **我们胜** | 唯一场景，也是最高频场景 |
| 出租房不能打孔布线 | **我们胜** | 浏览器打开就完事 |
| 多只宠物不同房间 | **我们胜** | 旧手机放不同房间，零额外成本 |
| 出差/旅游时远程看宠物 | 摄像头胜 | 我们的设备在宠物那侧，主人远端只能看报告 |

**结论**：我们的产品不是"摄像头的替代品"，而是"摄像头的互补品——覆盖主人不在线时的盲区"。有摄像头的用户可以同时用我们，没摄像头的用户可以先用我们零成本试水。

## 3. 用户调研与需求分析

> **调研方法论**：采用三来源三角验证——行业报告（看大盘） + 小红书帖子分析（看真人语言） + 问卷调查（定量验证）。目的是交叉确认痛点的真实性和优先级，而非单一来源主观判断。

---

### 3.1 行业报告数据（来源1）

三份公开报告的核心提取：

**《2025中国宠物行业白皮书》（派读宠物）**

- 城镇宠物消费市场规模 3250 亿元，同比增长 8.7%
- 城镇养宠人数 7300 万，犬主人 5200 万
- 养狗人群中 26-35 岁占比 48%，是绝对主力
- 宠物智能设备渗透率仅 12.3%，远低于干粮（89%）、零食（67%）

**艾媒咨询《2024-2025中国宠物行业运行状况及消费行为监测报告》**

- 宠物主人日均外出时长 9.2 小时
- "宠物独自在家时最担心的问题"：情绪焦虑/抑郁 41.3%、乱叫被邻居投诉 36.7%、拆家/破坏物品 22.0%
- 购买宠物摄像头的用户中，63% 表示"买了后悔，因为看了也做不了什么"

**京东《2025宠物消费趋势报告》**

- 宠物摄像头销量同比增长 27%，但智能设备仅占宠物消费总额的 2.1%
- 用户评论 Top 3 抱怨：① 安装布线麻烦 ② App 推送太多太烦 ③ 喊话功能不灵敏

**报告层面的交叉结论**：市场在增长但智能设备渗透极低，核心障碍不是"用户不需要"而是"现有方案解决不了问题"——63% 的后悔率说明摄像头满足了"看到"但没有满足"做到"。

### 3.2 小红书帖子洞察（来源2）

手动分析 200 条帖子（关键词："狗叫被邻居投诉""狗狗分离焦虑""宠物摄像头有用吗""上班狗自己在家"），深度编码 50 条，每条打 1-2 个标签。

**高频帖子类型和用户原话：**

> **帖子 A**（426 评论）："刚搬进新公寓第三天，邻居在门口贴了张纸条说我家狗叫了一下午。真的不知道它会叫成这样，平时我在家它超级乖的。"

> **帖子 B**（188 评论）："每次看到业主群有人@我就心跳加速，一定又是我家狗……装了摄像头确认了就是嚎，但是开摄像头看有啥用啊我看着它嚎然后干着急吗"

> **帖子 C**（256 评论）："求推荐有效防止狗叫的方法，已经试过：1. 出门前遛一小时 2. 漏食球 3. 放音乐 都不管用。邻居说再有下次就报警"

> **帖子 D**（野生验证型）："突发奇想给我家猫录了我的声音然后出门前放着循环播放，回来发现它居然没躲床底下！这是不是说明它听到我声音比较安心？？"

**标签编码统计（50 条深度分析样本）：**

| 标签 | 频次 | 说明 |
|------|------|------|
| 狗叫被投诉 | 31 | 最高频，社交压力是核心驱动力 |
| 看了没用 | 26 | 对摄像头的普遍失望 |
| 求助/求推荐 | 22 | 用户主动寻找方案，市场有需求 |
| 心疼/亏欠 | 18 | 情感诉求，"觉得对不起狗" |
| 土办法自创 | 7 | 用户已经在自己做"主人语音安抚" |
| 多宠物差异 | 5 | 需要独立管理，不是一刀切 |

**关键交叉发现**："狗叫被投诉" ∩ "看了没用" = 14 条（28%）。这些用户同时遇到两个问题——狗叫且现有方案无效，是我们的核心目标用户。

**核心洞察**：帖子 D 是"野生验证"——用户在没有产品的情况下，用土办法试了"主人语音安抚"这个假设，并且初步验证了。我们的产品就是这个土办法的工业化版本。

### 3.3 问卷调查（来源3）

**投放渠道和回收：**

| 渠道 | 为什么选这个渠道 | 回收 |
|------|-----------------|------|
| 养狗微信群（柯基/柴犬/金毛/泰迪各1个） | 品种群用户活跃，狗叫问题是高频话题 | 25 份 |
| 小红书私信发过分离焦虑帖的用户 | 精确触达已验证痛点的人群 | 8 份 |
| 豆瓣"养狗劝退小组" | 极端案例（已被投诉/想弃养），能听到最真实的后果 | 8 份 |
| 朋友圈/同事转介绍 | 补充非核心养宠圈样本，避免群内样本同质化 | 12 份 |
| **总计** | | **53 份（有效 47 份）** |

**问卷结构和每道题的逻辑：**

```
Q1. 年龄范围 → 验证白皮书"26-35岁主力"结论
Q2. 居住情况 → 验证假设：合租/独居对狗叫更敏感
Q3. 宠物类型 → 后续交叉分析用
Q4. 每天宠物独处时长 → 8小时以上是核心使用场景
Q5. 最担心什么（多选Top2）→ 核心痛点排序
Q6. 目前用什么方式应对 → 竞品渗透率 + 替代方案分布
Q7. 没用摄像头的原因 → 关键题：验证"觉得没用"vs"太贵"谁是主因
Q8. 产品概念接受度 → 直接验证"你会不会试试"
Q9. 功能排序（拖拽排序）→ 指导 MoSCoW 优先级

漏掉但后续补充的两道关键题：
Q+. 您家里有没有闲置的智能手机/平板（能开机联网但不当主力机用）？
Q++. 如果没有，是否愿意花 100-200 元买一台旧手机放家里专门跑这个？
```

**核心数据结果：**

```
▸ 年龄：26-30岁 38% | 31-35岁 34% | 18-25岁 19% | 36+ 9%
▸ 居住：合租 43% | 独居 30% | 家庭自住 27%
▸ 独处时长：8-10h 55% | >10h 21% | 4-8h 19% | <4h 5%
▸ 最担心（多选Top2）：
  ① 乱叫被投诉 68%  ② 情绪焦虑 51%  ③ 拆家 28%  ④ 健康 15%

▸ 现有方案：
  什么都没做但担心 43% | 放音乐/循环录音 26%
  宠物摄像头 11%     | 漏食球/玩具 13%

▸ 不买摄像头的原因（未购买者）：
  觉得看了也做不了什么 40% | 太贵 31% | 租房不方便 18% | 不了解 11%

▸ 产品概念接受度：非常愿意 36% + 可以试试 49% = 85% 愿意尝试

▸ 功能排序（加权分，5 分制）：
  ① 自动检测+播放安抚语音 4.3  ② 叫声检测+通知 3.8
  ③ 每日情绪报告/趋势 3.4     ④ 多房间多宠物管理 2.9
  ⑤ 历史数据查看 2.7
```

**关键交叉分析：**

居住情况 × 最担心狗叫被投诉：

| 居住情况 | 最担心狗叫被投诉 |
|---------|----------------|
| 合租 | **84%** |
| 独居 | 61% |
| 家庭自住 | 31% |

→ 合租用户对狗叫的敏感度是家庭自住用户的 2.7 倍。**核心目标用户不是泛泛的"养狗的人"，而是合租养狗人。**

独处时长 × "看了没用"占比：

| 每日独处时长 | "觉得摄像头没用"占比 |
|------------|-------------------|
| >10 小时 | **53%** |
| 8-10 小时 | 38% |
| <8 小时 | 18% |

→ 视频监控的价值随独处时间增加而递减。自动安抚的价值随独处时间增加而递增。

| 产品态度 | "零成本可以试试" | 问卷 85% 有兴趣 |

**次要用户：多宠物家庭/出差人士**

| 维度 | 描述 | 来源 |
|------|------|------|
| 宠物数量 | 2+ | 问卷 22% + 小红书 5 条 |
| 核心痛点 | 出差时多只宠物各自状态不明 | 小红书上"出差回来猫瘦了一圈" |
| 差异化价值 | 每个房间放一台旧手机，零额外成本 | 摄像头需买多台 |

### 3.5 设备持有与用户分层（调研修正）

**一个必须承认的局限**：原始问卷漏掉了"用户是否有闲置设备"这道题。这是调研设计中的一个疏漏——整个产品逻辑建立在"闲置设备复用"上，但没有验证前提条件。

后续通过补充调查（同一渠道追加）修正了这个数据：

| 层级 | 描述 | 占比（估算） | 对我们的价值 |
|------|------|------------|------------|
| **A 层** | 有闲置手机/平板（抽屉里有一台备用机） | ~30% | 核心种子用户，零成本试用 |
| **B 层** | 没有闲置设备，但愿意花 100-200 买二手手机 | ~35% | 比 Furbo（¥500+）便宜，可转化 |
| **C 层** | 没有闲置设备，也不想为这个专门买 | ~35% | 当前不可触达，需智能音箱方案覆盖 |

**A 层 × 合租 × 核心痛点 = 最优质的种子用户。** 他们同时满足三个条件：有设备、有痛点、零决策成本。

**产品策略随分层调整：**

| 阶段 | 目标层 | 方案 |
|------|--------|------|
| 短期（MVP） | A 层 | 免费 App + 闲置设备，验证产品闭环 |
| 中期 | B 层 | 推出 ¥99 ESP32 硬件套装，覆盖"愿意付费但不想买摄像头"的用户 |
| 长期 | C 层 | 接入智能音箱（小爱/天猫精灵），利用已有设备 |

**关于"零硬件成本"叙述的修正**：产品的真实成本结构不是"零"，而是：
- A 层用户：真实成本 ≈ 0（闲置设备本来就在）
- B 层用户：真实成本 ≈ ¥100-200（二手手机）
- C 层用户：当前不可触达

面试中不能再说"零硬件成本"，而应该说"硬件决策分离——用户可以用已有设备先体验，满意后再决定是否购买专用设备"。

### 5.4 校准引导页（核心新增）

**页面路径：** `/calibrate/:petId?mode=guard|mark`

**不直接显示的设置项**：校准流程完全隐藏 dB 数字。用户只看到引导文案和进度条。

**第一步：环境基线采样（10 秒）**

```
┌──────────────────────────────────┐
│  ← 取消校准                      │
│                                  │
│         🎤                      │
│    正在检测环境底噪...            │
│    请保持房间安静                │
│                                  │
│    ████████████░░░░  8/10 秒    │
│                                  │
│    提示：正常的环境声音（窗外的    │
│    车流声、空调声）都会被计入基线  │
│    所以请按宠物平时独处时的环境    │
│    来设置（关窗/开窗、开空调等）  │
└──────────────────────────────────┘
```

**技术细节**：10 秒内每秒采样一次 RMS 值，取均值作为 E_base。采样期间如果有突发巨响（超过均值 2 倍），该帧丢弃不参与均值计算。

---

**第二步：宠物叫声采样（15 秒，可跳过）**

```
┌──────────────────────────────────┐
│                                  │
│         🐕                      │
│    请让宠物发出叫声              │
│   （或播放一段宠物叫的视频）      │
│                                  │
│    ██████░░░░░░░░░░  5/15 秒    │
│                                  │
│    📊 检测到叫声峰值：—— dB      │
│                                  │
│    [ 跳过这一步，用默认值 ]      │
│                                  │
│    提示：如果宠物暂时不叫，       │
│    可以跳过，系统会根据体型       │
│    自动设定一个合理的默认值       │
└──────────────────────────────────┘
```

**技术细节**：15 秒窗口内持续记录 RMS 值的 top-3 峰值，三者取平均作为 P_peak。排除单次偶然尖峰（如主人不小心碰了麦克风）。

**完成页：**

```
┌──────────────────────────────────┐
│                                  │
│         ✅                      │
│    校准完成！                    │
│                                  │
│    你的环境基线：E_base          │
│    你的宠物叫声：P_peak          │
│    触发阈值：T = (E_base+P_peak)/2│
│                                  │
│    关于这个阈值：                 │
│    · 如果之后误触发太多 → 微调 +2 │
│    · 如果狗叫了没触发 → 微调 -2  │
│                                  │
│    [ 开始守护 ]  [ 开始标记 ]    │
│    [ 微调 ±2dB ][ 微调 ±5dB ]   │
└──────────────────────────────────┘
```

**阈值计算规则：**

| 条件 | 阈值 T | 说明 |
|------|--------|------|
| 完成第一步 + 第二步 | E_base + (P_peak - E_base) × 0.5 | 环境到叫声的中点，最大化检测同时最小化误报 |
| 只完成第一步（跳过第二步） | E_base + 体型差值 | 小 +10 / 中 +15 / 大 +20 |
| 两步都失败 | 默认相对值 +15 | 最后的后备 |

**校准数据存储**：校准结果存 Supabase `pet_calibrations` 表，包含 `pet_id`、`E_base`、`P_peak`（可为 null）、`threshold`、`calibrated_at`。下次开启守护时自动读取最近一次的阈值。

### 5.5 守护模式运行中页

**页面路径：** `/guarding/:petId`

**核心改造**：从"两个状态（正常/触发）"变成"显式状态机 + 彩色波形 + 自动麦克风管理"。

**状态机流转：**

```
 Monitoring（监听中）
     │
     ├─ 滑动窗口触发 ──→ Triggered（触发！）
     │                      │ 暂停麦克风输入
     │                      │ 播放安抚语音
     │                      │ 显示红色横幅
     │                      ↓
     │                   Cooldown（冷却中）
     │                      │ 等待 60 秒
     │                      │ 显示"冷却中"倒计时
     │                      │ 恢复麦克风输入
     │                      ↓
     └──────────── 回到 Monitoring ←──┘
```

**每个状态的 UI 表现：**

| 状态 | Emoji | 横幅 | 波形 | 按钮 |
|------|-------|------|------|------|
| Monitoring | 😴 脉冲动画 | 无 / "守护中" | 绿/黄区域实时跳动 | [停止守护] |
| Triggered | 😱 弹跳动画 | 🔴 "检测到叫声！正在播放安抚语音" | 冲入红色区，剧烈跳动 | [停止守护] 灰色不可点 |
| Cooldown | 😌 轻微动画 | 🟡 "冷却中… 45 秒后恢复监听" | 回到绿色区，安静跳动 | [停止守护] |

**彩色波形区域（GuardingView 的 Waveform 组件改造）：**

```
Canvas 波形图上叠加三条颜色区：

🔴 红色区（超出阈值）}
🟡 黄色区（接近阈值）} ──→ 阈值线
🟢 绿色区（环境范围）

实时效果：
- 安静时，波在绿色区跳动
- 有人说话/电视声，波偶尔碰到黄色区
- 狗叫时，波冲进红色区
- 冲进红色区 → 滑动窗口开始计数
```

**滑动窗口触发判定（替代旧的持续时长判定）：**

```
不是在代码里写：
  if (db > threshold && duration > 2000ms) → trigger

而是：
  维护一个长度为 5 秒的环形缓冲区
  每秒采样 10 次（即 50 个采样点）
  如果 5 秒内有 ≥ 3 个"尖峰"（超过阈值的采样点）
  → 触发

  尖峰的定义：
    一次尖峰 = 连续 ≥ 3 个采样点超过阈值
    （即至少 0.3 秒的持续超阈值，排除瞬时尖峰）

  触发后：
    1. 暂停 AnalyserNode（断开输入，防止录到安抚语音）
    2. 随机选一条安抚语音 → Audio 元素播放
    3. 播放结束后 → 进入 Cooldown
    4. Cooldown 结束 → 恢复 AnalyserNode → 回到 Monitoring
```

**退出守护 → 跳转报告页：**

```
[停止守护] → 确认弹窗 → 结束会话
  → 记录 end_time + anxiety_events 统计
  → 如果是标记模式结束，提醒"你还有 N 条未标记的声音，去声音日志看看"
  → 调用 DeepSeek API 生成日报
  → 跳转 /report/:sessionId
```

**麦克风自动管理规则：**

| 状态切换 | 麦克风操作 | 原因 |
|---------|-----------|------|
| 进入 Monitoring | 启动 AnalyserNode | 开始监听 |
| Monitoring → Triggered | 断开 AnalyserNode 输入 | 防止录到主人自己的安抚语音形成死循环 |
| Triggered → Cooldown | 保持断开 | 播放刚结束，宠物可能还在反应 |
| Cooldown → Monitoring | 重新连接 AnalyserNode | 冷却结束，恢复监听 |
| 任何状态 → 停止 | 关闭 AudioContext + Stream | 完全释放资源 |

### 5.6 标记模式运行中页

**页面路径：** `/marking/:petId`

标记模式与守护模式的核心区别：**触发后不播放安抚语音，而是录制音频片段存储到 Supabase。**

**UI 表现：**

```
┌──────────────────────────────────┐
│  ← 退出标记模式                  │

│       🎙️ 标记模式运行中          │

│       实时波形（绿/黄/红）        │
│                                  │
│     📊 本次已记录：3 段声音       │

│                                  │
│     [ 停止标记并查看日志 ]        │
```

**技术实现**：

触发判定逻辑与守护模式完全一致（同一套状态机 + 滑动窗口），不同点在于 Triggered 状态的行为：

```
守护模式 Triggered：
  → 暂停麦克风 → 播放安抚语音 → 冷却 → 恢复

标记模式 Triggered：
  → 不暂停麦克风 → 用 MediaRecorder 录制当前片段（前后各 3 秒，共 6 秒）
  → 保存 WebM blob → /tmp/（本地先存 .webm）
  → 上传到 Supabase Storage: audio_snippets/{pet_id}/{timestamp}.webm
  → 在 audio_snippets 表插入一条记录（peak_db, recorded_at, audio_url, label=null）
  → 直接进入冷却（无安抚播放）
  → 冷却结束 → 恢复判定
```

```

│                                  │
│  本次记录了 5 段声音              │
│  📊 峰值范围：52 ~ 71 dB         │
│                                  │
│  [ 去声音日志标记 ]              │
│  [ 稍后再说 ]                   │

```

## 6. 音频检测引擎技术方案

### 6.1 技术选型

| 环节 | 技术 | 说明 |
|------|------|------|
| 音频采集 | `navigator.mediaDevices.getUserMedia` | 浏览器原生，无需插件 |
| 实时分析 | `AudioContext` + `AnalyserNode` | FFT 256 点，时域（`getByteTimeDomainData`） |
| 分贝计算 | RMS（均方根）相对值 | `20 × log₁₀(RMS)`，不做绝对 dB SPL 校准 |
| 音频录制 | `MediaRecorder` API | 标记模式录制 6 秒 WebM 片段 |
| 语音存储 | Supabase Storage | `audio_snippets` bucket，WebM 格式 |
| 语音播放 | `Audio` 元素 | URL 直链播放 |
| 数据持久化 | Supabase PostgreSQL | `audio_snippets` 表 + `pet_calibrations` 表 |

### 6.2 核心原则：相对值优于绝对值

Web Audio API 的 `getByteTimeDomainData` 返回 0-255 量化值，RMS 计算后得到的 dB 值是**相对于设备麦克风灵敏度**的值。同一只狗叫，在 iPhone 上可能显示 85，在低端安卓上可能显示 55。

因此，所有阈值策略都基于**"高于环境基线多少"**而非绝对 dB。

### 6.3 动态基线校准算法

**环境基线采样（10 秒）：**

```
设置：采样率 10 Hz（每秒 10 次）
总采样点：100 个

对于每个采样点：
  1. 从 AnalyserNode 读取 timeData（128 个 uint8）
  2. 计算 RMS = sqrt(mean((x - 128)² / 128²))
  3. 计算 dB_rel = 20 × log₁₀(max(RMS, 1e-10))

异常值过滤：
  如果某个采样点的 dB_rel 超过已有均值的 2 倍：
  → 丢弃该采样点（排除偶然巨响，如主人咳嗽）

最终：
  E_base = mean(有效采样点)
```

**宠物叫声采样（15 秒）：**

```
采样率同上。
记录所有采样点的 dB_rel。
取 top-3 峰值的均值作为 P_peak。

目的：排除单次偶然尖峰（如主人碰到麦克风）。
3 次均值更能代表"这只狗叫起来通常多响"。

最终：
  如果完成第二步 → threshold = E_base + (P_peak - E_base) × 0.5
  如果跳过第二步 → threshold = E_base + 体型差值
  如果两步都失败 → threshold = E_base + 15
```

**阈值系数 0.5 的数学依据**：信号检测理论（Tanner & Swets, 1954）的最优决策边界原理——在噪声分布 N(E_base, σ²) 和信号分布 N(P_peak, σ²) 之间取中点，最大化检测概率同时最小化虚警概率。这是一种最大化 ROC 曲线下面积（AUC）的朴素策略。

### 6.4 滑动窗口防抖算法

**设计动机**：狗叫是间歇性短声（单次 0.5-2 秒，Overall 1997），单个长音峰更容易是关门/喇叭。多个短音峰在短时间窗口内出现，才是狗叫的标志。

**算法实现：**

```
windowSizeMs = 5000          // 5 秒滑动窗口
spikeThresholdCount = 3      // 窗口内 3 次尖峰即触发
sampleIntervalMs = 100       // 每 100ms 采样一次（10 Hz）
minSpikeDuration = 3         // 尖峰 = 连续 ≥ 3 采样点超阈值（0.3 秒）
                              // 排除瞬时尖峰（掉东西）

维护一个环形缓冲区 spikeBuffer，存储最近 50 个采样点（5 秒 × 10 Hz）

每个采样点：
  if (db_rel > threshold) → value = 1
  else → value = 0

尖峰检测（游程编码）：
  扫描 spikeBuffer，找出所有连续 1 的游程
  游程长度 ≥ 3 的 = 一个有效的"尖峰"
  统计窗口内的有效尖峰数量

判定：
  if (有效尖峰数 ≥ spikeThresholdCount) → 触发！
```

**为什么不使用频域过滤**：当前 FFT 256 点的频率分辨率 = 采样率 44100 / 256 ≈ 172 Hz/频点。狗的叫声主要能量在 200-1000 Hz（中型犬）或 800-3000 Hz（小型犬），256 点 FFT 在这个范围内只有 5-12 个频点，不足以进行可靠的品种或声音类型分类。后续可选升级为 2048 点 FFT（分辨率 21.5 Hz/频点），但会增加 CPU 负载。当前 MVP 用滑动窗口替代频域过滤。

### 6.5 状态机设计

```
┌──────┐   startCalibration   ┌──────────────┐
│ IDLE │ ─────────────────→   │ CALIBRATING  │
└──────┘                      └──────┬───────┘
                                     │ calibrationDone
                                     ↓
                              ┌──────────────┐
                        ┌──→  │ MONITORING   │
                        │     └──────┬───────┘
                        │            │ slidingWindowTriggered
                        │            ↓
                        │     ┌──────────────┐
                        │     │ TRIGGERED    │
                        │     │ (暂停麦克风)  │
                        │     └──────┬───────┘
                        │            │ soothingPlaybackFinished
                        │            ↓
                        │     ┌──────────────┐
                        │     │ COOLDOWN     │
                        │     │ (60s 倒计时)  │
                        │     └──────┬───────┘
                        │            │ cooldownExpired
                        └────────────┘
```

**每个状态的进/出动作：**

| 状态 | 进入时 | 退出时 |
|------|--------|--------|
| IDLE | — | — |
| CALIBRATING | 启动 AudioContext + 采样定时器 | 关闭采样定时器，保留 AudioContext |
| MONITORING | 连接 AnalyserNode，启动 requestAnimationFrame loop | 清空 spikeBuffer |
| TRIGGERED | **断开 AnalyserNode 输入**（防死循环），随机选语音，播放 Audio，记录 anxiety_event | 停止 Audio 播放 |
| COOLDOWN | 启动 60s 定时器，显示倒计时 | 恢复 AnalyserNode 输入 |

**TRIGGERED 状态下断开 AnalyserNode 的技术实现**：不销毁 AudioContext，而是将 MediaStreamSource 的 connect 目标从 AnalyserNode 改为一个哑节点（或直接 disconnect），播放结束后重新 connect 到 AnalyserNode。

### 6.6 标记模式录制实现

```
MediaRecorder 初始化：
  mimeType: 'audio/webm'（Chrome 原生支持）
  audioBitsPerSecond: 64000（64 kbps，6 秒 ≈ 48 KB）

触发后的录制流程：
  1. 环形缓冲区保留最近 3 秒的音频数据
  2. 触发后，继续录制 3 秒
  3. 拼接为 6 秒 WebM blob
  4. 上传到 Supabase Storage：audio_snippets/{pet_id}/{timestamp}.webm
  5. 在 audio_snippets 表插入记录
  
Supabase Storage bucket 结构：
  audio_snippets/
    {pet_uuid}/
      2026-05-23T14-35-22.webm
      2026-05-23T15-02-11.webm
```

## 8. 情绪评估体系设计

### 9.1 为什么需要情绪评估

宠物不会说话，主人无法直观了解宠物独处时的情绪状态。情绪评估体系将音频数据转化为可量化的情绪指标，帮助主人：
- 了解宠物分离焦虑的严重程度和变化趋势
- 评估安抚策略的有效性
- 发现隐藏的行为模式（如特定时段焦虑高发）

### 9.2 核心评估指标

| 指标ID | 指标名称 | 定义 | 计算公式 | 数据来源 |
|--------|---------|------|---------|---------|
| E01 | 焦虑频次 | 单位时间内的焦虑触发次数 | 焦虑事件数 / 守护时长(小时) | anxiety_events |
| E02 | 峰值强度 | 当日最高分贝值 | MAX(peak_db) | anxiety_events |
| E03 | 焦虑总时长 | 当日焦虑状态累计时长 | SUM(duration_sec) / 60 (分钟) | anxiety_events |
| E04 | 高发时段 | 焦虑事件最集中的时间段 | MODE(hour) | anxiety_events.triggered_at |
| E05 | 安抚介入率 | 每次焦虑是否都有语音响应 | 有语音播放的事件数 / 总事件数 | anxiety_events.voice_played_id |
| E06 | 情绪趋势 | 与昨日的焦虑总时长变化 | (昨日焦虑时长 - 今日焦虑时长) / 昨日焦虑时长 | daily_reports.stats_json |

## 10. 风险与假设

### 10.1 关键假设

| 假设 | 风险等级 | 验证方式 | 当前状态 |
|------|---------|---------|---------|
| 用户有闲置设备（手机/电脑）可长时间开浏览器 | 中 | 补充问卷调查（Q+/Q++），随后种子用户验证 | 设备分层已纳入画像，覆盖 A+B 层 |
| 宠物叫声能被相对分贝阈值有效检测 | 中 | 需在不同环境（安静/嘈杂）+ 不同宠物体型测试 | 滑动窗口 + 动态基线已大幅降低此风险 |
| 主人预录语音能有效安抚宠物 | 中 | 需种子用户反馈验证 | 趋势图（阶段三）是此假设的验证工具 |
| 校准第二步用户可以配合完成 | 高 | 需要实际可用性测试 | 设计了跳过机制（按体型给默认值），不会卡死 |
| 标记模式下用户愿意回听和标记 | 中 | 需要种子用户行为数据 | 系统建议阈值是这个行为的激励——标记越多、建议越准 |
| DeepSeek API 持续可用 | 低 | — | 有离线降级模板 |

### 10.2 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 浏览器休眠导致监听中断 | 核心功能不可用 | 使用 Wakelock API 保持唤醒（后续版本）；当前依赖用户保持设备前台 |
| 麦克风权限被拒绝 | 无法使用 | 校准引导中清晰说明权限用途；拒绝后提供手动输入阈值选项（列表项选宠物体型+环境描述） |
| 不同设备麦克风灵敏度差异巨大 | 基线校准结果不可迁移 | 每次换设备必须重新校准。校准页面提示"如果换了设备，请重新校准" |
| 滑动窗口参数（5秒/3次）不适应所有宠物 | 部分宠物漏报或误报 | 设置页开放窗口长度和冷却时间调整 |
| MediaRecorder 兼容性（非 Chrome 浏览器） | 标记模式不可用 | 检测 `MediaRecorder` API 可用性，不支持时标记模式灰色不可选 |
| Supabase Storage 上传失败（网络断） | 录音片段丢失 | 本地 IndexedDB 临时缓存，网络恢复后补传 |

### 10.3 产品风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 校准流程（两步 25 秒）可能劝退用户 | 首次体验流失 | 第二步设计为可跳过；提供"10 秒快速校准"选项（仅测环境，用体型默认值） |
| 主人语音可能让宠物更焦虑 | 产品效果适得其反 | 语音录制页提示"请用平时在家和它说话的语气，不要用激动的语气" |
| 标记模式录到 50 条未标记 → 回听成本太高 | 用户放弃标记 | 只显示最近 20 条；超过 20 条时折叠旧记录，顶部提示"你还有 XX 条未标记" |
| 宠物脱敏（对语音习惯后无效） | 长期效果下降 | 随机播放多条语音 + 鼓励定期更新语音 + 趋势图可检测"安抚成功率下降"信号 |

## 11. 附录：术语表

| 术语 | 英文 | 定义 |
|------|------|------|
| 守护会话 | Guard Session | 一次从"开启守护"到"停止守护"的完整监听周期 |
| 守护模式 | Guard Mode | 检测到叫声 → 自动播放安抚语音 → 冷却 → 循环 |
| 标记模式 | Marking Mode | 检测到异常声音 → 录制片段 → 上传 → 用户回家回听标记 |
| 校准 | Calibration | 首次使用时的两步引导流程：测环境基线 + 测宠物叫声峰值 |
| 环境基线 | E_base / Environmental Baseline | 10 秒静默采样期间的环境 RMS 均值（相对分贝） |
| 宠物叫声峰值 | P_peak / Pet Vocalization Peak | 15 秒叫声采样期间的 top-3 尖峰均值 |
| 滑动窗口 | Sliding Window | 5 秒环形缓冲区，统计窗口内的"尖峰"数量 |
| 尖峰 | Spike | 连续 ≥ 3 采样点（≥ 0.3 秒）超过阈值的一次事件 |
| 焦虑事件 | Anxiety Event | 一次被判定为"宠物在叫"的尖峰集合（滑动窗口触发） |
| 安抚语音 | Soothing Voice | 用户预录的用于安抚宠物的音频片段 |
| 声音片段 | Audio Snippet | 标记模式下录制的 6 秒 WebM 音频（触发前后各 3 秒） |
| RMS | Root Mean Square | 均方根，用于计算音频信号的有效能量值。公式：√(Σ(x-128)²/N) |
| 相对分贝 | dB (relative) | `20 × log₁₀(RMS)`，相对于设备麦克风灵敏度的分贝值，不做绝对 SPL 校准 |
| 阈值 | Threshold | 触发焦虑事件的音量临界值 = E_base + Δ |
| 冷却时间 | Cooldown | 两次安抚之间的最小间隔（默认 60 秒），给宠物平复情绪的时间 |
| 情绪日报 | Daily Report | 基于当天守护数据由 AI 生成的文字总结 |
| 情绪趋势 | Emotion Trend | 多周焦虑数据的变化趋势（上升/下降/平稳） |
| 阈值建议 | Threshold Suggestion | 基于标记数据（最低狗叫声 - 最高噪音）/ 2 的建议值 |
| 状态机 | State Machine | 5 状态流转引擎（Idle → Calibrating → Monitoring → Triggered → Cooldown） |
