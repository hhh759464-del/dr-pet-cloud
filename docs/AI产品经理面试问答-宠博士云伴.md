# 宠博士·云伴 — AI产品经理面试问答手册

---

## 一、项目概述

### Q1：用一句话介绍你的项目

> 宠博士·云伴是一个 Web 端的宠物分离焦虑智能安抚产品。主人出门前把闲置手机打开浏览器放在宠物旁边，系统自动监听环境音，检测到宠物吠叫时自动播放主人预录的安抚语音，结束后 AI 生成情绪日报。零硬件成本，浏览器打开即用。

---

### Q2：你们要解决什么用户问题？

> 城市上班族养宠比例很高，但白天在外面上班，不知道宠物是不是在家里叫了一整天。邻居投诉、房东警告都很常见。市面上能看宠物的摄像头要 ¥300-1000，而且只能看不能管，看到狗叫了你也只能干着急。
>
> 我们的产品核心就是两件事：**自动发现焦虑 + 自动介入安抚**。不是让主人盯着监控看，而是让闲置设备替主人陪伴宠物。

---

### Q3：目标用户是谁？

> 核心用户是城市独居/合租的上班族，养狗，有分离焦虑问题，租房不太方便打孔布线装摄像头。次要用户是多宠物家庭，出差时想照看多只宠物。
>
> 中国城镇养宠人群超过 7000 万，独居和合租上班族占比超过 60%，约 40% 的宠物犬有不同程度的分离焦虑。

---

### Q4：跟市面上的宠物摄像头有什么区别？

> | 维度 | 传统摄像头 | 宠博士·云伴 |
> |------|----------|-----------|
> | 硬件成本 | ¥200-1000 | 0（闲置设备复用） |
> | 安装方式 | 插电配网打孔 | 浏览器打开即用 |
> | 安抚方式 | 被动观看/手动喊话 | 自动检测→触发→播放主人语音 |
> | 数据报告 | 无 | AI 情绪日报+趋势分析 |
> | 多宠物 | 通常不支持 | 原生支持 |

---

### Q5：为什么做 Web 而不是 App？

> MVP 阶段的核心考量是降低使用门槛。做 App 需要下载安装、上架审核、 iOS 和 Android 两套代码。Web 一个链接就能用，而且我们依赖于 Web Audio API，浏览器对麦克风的支持已经足够好了。
>
> 代价是后台标签页可能被系统挂起，这个在路线图里用 Wake Lock API 解决。长远来看如果用户量起来，App 版本是必然的。

---

### Q6：项目的技术栈是什么？

> 前端 Vue 3 + Vite + Tailwind CSS，后端用的是 Supabase BaaS（自带用户认证、PostgreSQL 数据库、文件存储），没有自己写后端服务器。AI 部分接的是 DeepSeek Chat API 用来生成情绪日报。音频处理全部用浏览器原生的 Web Audio API 和 MediaRecorder API，不依赖任何第三方音频库。

---

### Q7：为什么用 DeepSeek 而不是 GPT 或 Claude？

> MVP 阶段选 DeepSeek 三个原因：第一是成本，DeepSeek 的价格远低于 GPT-4；第二是中文能力强，我们的情绪日报需要温柔亲切的中文表达，DeepSeek 在中文本土化上有优势；第三是我们做了降级策略，API 挂了有本地模板兜底，不绑定任何一家。后续可以根据效果切换模型，不影响用户体验。

---

## 二、核心功能与逻辑

### Q8：音频监测到安抚播放的完整流程是怎样的？

> 用户点"开启守护" → 浏览器弹窗请求麦克风授权 → 授权后系统开始持续监听环境音。
>
> 每秒检测 60 次，实时计算当前的分贝值。如果分贝值连续 2 秒超过 75dB（默认值），系统判定"宠物在持续大声叫"，触发焦虑事件：
> - 屏幕上 emoji 从 😴 变成 😱，弹出红色警告条
> - 从数据库随机选一条主人预录的安抚语音，播放出来
> - 记录这次焦虑事件到数据库
> - 进入 60 秒冷却期，冷却期内不重复触发
>
> 主人回家后点"停止守护"，系统汇总所有焦虑事件，调用 AI 生成情绪总结。

---

### Q9：触发判断的逻辑是怎么设计的？

> 不是简单的一超过阈值就报警，那是噪音检测器。我们的触发引擎是一个**四状态单向循环**：
>
> 1. **IDLE**（空闲监听）— 正常状态
> 2. **LISTENING**（接近阈值）— 声音超 75dB 但还没持续 2 秒
> 3. **TRIGGERED**（触发）— 确认焦虑，播放语音，记录事件
> 4. **COOLDOWN**（冷却）— 60 秒内不再触发，避免反馈循环
>
> 每个状态只有一种出口，不允许互相跳转。这样保证了逻辑清晰，不会出现"一边放语音一边又触发新事件"的 bug。

---

### Q10：AI 情绪日报是怎么生成的？

> 守护结束后，系统把这次会话的数据打包——守护多长时间、焦虑触发了几次、最高多大声、发生在什么时段、对比昨天是多了还是少了——作为一个结构化 prompt 发给 DeepSeek。
>
> Prompt 的角色设定是"一位温柔的宠物情绪分析师"，约束是 50 字左右、语气亲切像跟老朋友聊天、有焦虑要先共情再给正向引导。DeepSeek 返回自然语言总结。
>
> 如果 API 不可用，有本地模板降级方案，用户永远能看到结果。

---

### Q11：如果用户没录安抚语音怎么办？

> 触发时会播放系统生成的默认提示音（一个三音阶的正弦波），同时在页面上提示用户"还没有录制安抚语音，建议去录音"。这是一个软引导，降低新用户的初次使用门槛。
>
> 长期来看，可以根据宠物的品种/类型匹配默认的专业安抚音频，作为冷启动方案。

---

## 三、遇到的困难与解决方案（面试重点）

### 困难一：怎么区分狗叫和环境噪音？

> Web Audio API 只能告诉你"麦克风收到了多大的声音"，分不清是狗叫还是关门声还是外面过车。
>
> 我做了一个三层防护的触发策略：
>
> **第一层——持续确认**：不是声音一大就报警，要求连续超过阈值 2 秒才触发。一声"砰"的关门瞬间就过去了，不会被误判。
>
> **第二层——冷却机制**：安抚语音播放后强制 60 秒冷却期，避免安抚语音被麦克风重新收进去形成误触发循环。
>
> **第三层——用户可调阈值**：40-100dB 滑块让用户根据自己宠物体型和环境调。因为大狗和博美的叫声音量差很远，不能一个阈值吃所有。
>
> 同时诚实地在产品文档里写了局限——"仅基于分贝，无法区分狗叫和其他高分贝声音"，不假装 100% 准确。

### 困难二：Web Audio API 拿到的不是真实分贝，怎么校准？

> 浏览器 AnalyserNode 返回的是 0-255 的原始采样值，不是物理世界的 dB 数。我查资料拼了一个近似公式 `20×log10(rms)+94`，但 94 这个偏移量是标准声压参考值，不同手机麦克风灵敏度差异很大。同一只狗叫，iPhone 和红米测出来的结果可能差 6dB 以上。
>
> 我的处理方式：
> 1. 用自己手机在不同距离（1m/3m/5m）播放狗叫测试，验证公式在常见场景下大致准确
> 2. 确定了"75dB 默认可覆盖大部分中小型犬叫"的基准
> 3. 把阈值做成用户可配置的滑块，把最终判断权交给实际使用者
> 4. 诚实承认这是近似值，不做过度宣称

### 困难三：守护中四个状态的切换逻辑怎么设计？

> 守护页面看起来很简单——就一个 emoji 加一个波形图。但背后有四个状态在互跳：空闲监听 → 接近阈值 → 触发安抚 → 冷却期 → 回到监听。
>
> 这几个状态之间有几个坑：
> - 冷却期内狗还在叫，要不要再触发？（不能，安抚白做了）
> - 播放语音时扬声器放出的声音被麦克风收进去怎么办？（会产生啸叫式反馈循环）
> - 用户在焦虑触发的瞬间点了"停止守护"怎么办？（状态机得干净退出）
>
> 我的解法是画了明确的状态机——每个状态只允许一种出口，不允许多种互跳。冷却期内 `aboveThresholdSince` 直接清空；播放语音时用 `isPlayingSoothing` 标志位暂停检测；停止守护时全部状态重置。这个状态机设计完之后，后续加功能（冷却期可配置、触发延迟可配置）改动量很小。

## 四、数据与指标

### Q12：你怎么衡量这个产品是有效的？

> 核心问题不是"功能跑没跑"，而是"宠物有没有真的被安抚好"。我设计了七个量化指标：
>
> - **焦虑频次**（每小时触发几次）— 衡量严重程度
> - **峰值强度**（最高叫多大声）— 衡量烈度
> - **高发时段**（什么时间段集中触发）— 帮主人发现模式
> - **安抚介入率**（触发后有没有放语音）— 系统响应完整度
> - **情绪环比**（和昨天比叫的时间多了还是少了）— 核心趋势指标
> - **月平静率**（一个月里多少天完全没叫）— 长期改善趋势
>
> 其中"情绪环比"是最关键的——如果连续多天数值在下降（焦虑减少），说明产品在起作用。如果反向增长，说明需要调整策略或者建议主人咨询兽医。

---

### Q13：怎么防止虚假数据？比如环境噪音被误判为焦虑事件？

> 这回到刚才说的三层防护——持续确认 + 冷却 + 可调阈值。另外产品文档里也写了一个诚实声明："仅基于分贝数据，无法 100% 确定宠物真实情绪状态。如有严重行为问题，建议咨询兽医行为专家。"
>
> 后续版本计划加入用户反馈机制——报告页底部有赞/踩按钮，可以让主人标注"这次触发确实是我家狗在叫"还是"这次是误触发"，用真实反馈来优化阈值模型。

---

### Q14：对比昨日的功能怎么做的？发现过什么问题？

> 生成今天日报时，多查一步：去数据库查昨天有没有 daily_report，有就把今天的焦虑次数跟昨天的对比，生成文字如"比昨天减少了 2 次焦虑，宝宝在进步"。
>
> 实际测试时发现过一个 bug：最开始这个对比只在前端页面显示了，但没真正存到数据库里，所以日历页和情绪分析页完全看不到对比数据。后来修复了——在报告生成的每一步（正常停止 × 跨午夜自动截断）都做了对比查询并存下来。

---

## 五、产品决策与取舍

### Q15：你做了哪些核心取舍？比如有什么功能是你故意不做的？

> 最大的取舍是**不做实时视频**。宠物摄像头的核心功能，但我们 MVP 不做。
>
> 原因：第一，WebRTC 视频对带宽和服务器成本有要求，跟我"零硬件成本"的定位矛盾；第二，有了视频会让用户停留在"看监控"的被动模式，而我想验证的是"主动安抚"这个核心假设——主人声音能不能安抚宠物。如果视频是核心卖点，产品就又退回了一个小号摄像头。先验证最核心的功能闭环，视频是有价值但非必要的增强。

---

### Q16：为什么不用机器学习直接识别狗叫声？

> MVP 阶段用的是分贝阈值，没有用 AI 做叫声分类。原因：
>
> 1. 数据获取难——需要大量标注的狗叫/猫叫/环境音数据集来训练模型
> 2. 浏览器端部署难——在浏览器里跑 TensorFlow.js 模型会增大加载体积和耗电
> 3. 先验证场景——核心假设是"自动检测+播放主人语音能安抚宠物"，在验证这个之前，分贝阈值够用
>
> 但这在路线图里，后续可以用频域分析 + 轻量模型来增强。

---

### Q17：如果产品要想继续突破，你觉得下一步最该做什么？

> 第一步肯定是**移动端适配 + Wake Lock**，让手机在后台也能持续监听。这是基础体验补课。
>
> 第二步，等积累了足够的种子用户数据后，可以做**频域分析**——把检测从"分贝阈值"升级为"频率加权"（狗叫主频 600-2000Hz），大幅降低误触发。
>
> 第三步，引入**用户反馈闭环**——让用户在日报里标注"这次触发是真的焦虑还是误触发"，用反馈数据持续优化检测准确率。长期可以做成个性化阈值，为每只宠物学习专属的触发模型。

---

### Q18：如果给你一个月时间重新做这个项目，哪些地方你会做得不一样？

> 1. **先做纸质原型/低保真测试**，而不是直接写代码。最开始花了不少时间在 UI 细节上，但其实应该先拿原型去跟养宠用户聊，验证"你愿不愿意把闲置手机开浏览器放在家"这个前置假设。
>
> 2. **第一天就考虑移动端**。虽然是 Web 端，但核心场景是手机，最开始没加 viewport 适配，后来补的。移动优先的布局应该从一开始就做。
>
> 3. **提前设计数据看板**。现在看运营数据要在 Supabase 控制台写 SQL，没有一个管理后台，每次想知道注册了多少用户、守护了多少次都要手动查。应该预留一个简单的数据概览页。

---

## 六、安全与隐私

### Q19：用户担心麦克风一直开着，隐私怎么保证？

> 1. 麦克风只在用户明确点"开启守护"时才请求权限，不是一打开网页就请求
> 2. 停止守护时立即释放麦克风，stream track 全部 stop
> 3. 录音流只做本地 RMS 换算，不上传任何原始音频到服务器
> 4. AI 日报传给 DeepSeek 的是结构化统计数据（数字），不是音频片段
> 5. 安抚语音文件存的是用户自己录的，只属于用户自己的 Supabase 项目

---

### Q20：用户数据安全怎么保障？

> 用了 Supabase 的 RLS（行级安全策略），每条数据入库时绑定了用户 ID，数据库层面保证只能查到自己的数据。前端的 API Key 是公开的 anon key，不暴露 service_role 管理密钥。DeepSeek 的 API Key 通过 Vite 编译时注入，不在前端暴露。
